## 1. État du projet

- Application **RestoPilote** = gestion de restaurant (React 19 + Express + Socket.io + PostgreSQL + Drizzle + Firebase Auth).
- Front **et** back sur **http://localhost:3000** via `npm run dev`. App fonctionnelle en local.

### Démarrer
```powershell
npm run dev        # front + back sur http://localhost:3000
npm run db:push    # (re)crée/synchronise les tables depuis le schéma Drizzle
npm run lint       # type-check TypeScript (doit finir sans erreur)
```

### Config locale (rappel)
- PostgreSQL base `chefsuite` (accès dans `.env` : `SQL_USER=user`, `SQL_PASSWORD=user`). Binaires : `C:\Program Files\PostgreSQL\16\bin`.
- `.env` : contient `ENABLE_TEST_LOGIN="true"` et `GEMINI_API_KEY` (clé **présente en local uniquement**, PAS sur GitHub — à reconfigurer sur une autre machine).
- Firebase : projet `chefsuite-local`, config dans `firebase-applet-config.json` (ignoré par git).
- ⚠️ `.env` et `firebase-applet-config.json` sont **ignorés par git** ; modèles fournis : `.env.example`, `firebase-applet-config.example.json`.
- ⚠️ `localhost` peut résoudre en IPv6 ; tester en ligne de commande avec `127.0.0.1`.

## 3. Ce qu'on a construit aujourd'hui (14/07)

**Commandes / additions**
- Champ **type** : sur place / à livrer. Champ **TVA** (prix TTC → ventilation HT/TVA/TTC, sur le ticket aussi).

**Livraisons** — nouvel onglet : client, tél, adresse, **date + heure**, **n° de commande** à livrer, **livreur** (rôle dédié). Apparaît dans le calendrier.

**Personnel** — rôle **Livreur** + rôle **Autre** (libre) ; import **CV** (PDF/image).

**Carte du Chef** — modale fiche technique **scrollable** (croix sticky). Ingrédients **liés au stock** (déduction auto), coût auto depuis le prix du stock.

**Stocks (Niveau 3)** — quantité **initiale vs restante**, **réapprovisionnement** (s'additionne), **historique d'appro daté**, **résumé** (compteurs + valeur totale). **Déduction auto** des ingrédients quand un plat passe à **« servi »** (idempotent). Alerte stock bas.

**Fournisseurs & Commandes** — séparés en 3 sous-onglets (Stocks / **Fournisseurs** annuaire / **Commandes** historique). Commande = **quantité × prix unitaire** ; **facture** (n° + photo). Payée → **dépense auto** (catégorie *Ingrédients*). **Reçue** → **alimente le stock** (+ mouvement d'historique + coût unitaire), avec possibilité de **créer l'article de stock à la volée**.

**Finances**
- Onglet **Rentrées d'argent** : additions payées (auto) + saisies manuelles (origine, moyen de paiement, justificatif…).
- **Dépenses** : catégorie dédiée **Ingrédients** (food cost juste) ; **charges récurrentes** mensuelles (génération auto).
- **Tableau de bord** : **ratios** (food cost, masse salariale, marge brute), **trésorerie 6 mois**, **meilleures ventes** (plats/boissons). Correction du **double-comptage** des fournisseurs (comptés une seule fois via les dépenses).

**Calendrier** — événements spéciaux **migrés en base** (fin du localStorage → persistent). Affichés aussi dans le **Calendrier de l'Établissement** du dashboard (avec livraisons).

**Paramètres** — **téléphone + adresse** (rattachés au propriétaire `dataOwnerId`).

**Divers** — onglets réordonnés (Tables → Carte du Chef → Réservations → Personnel) ; renommage **ChefSuite → RestoPilote** ; texte login « Gestion de Restaurant en ligne » ; IA testée OK (modèle **`gemini-flash-latest`**).

**Docs** — **`Guide-Utilisation-RestoPilote.docx`** (guide A→Z) ; **`installation.md`** mis à jour (clone GitHub + fichiers `.example`).

## 4. Schéma de la base (tables principales)

`users` (+ owner_id, restaurant_phone/address), `clients`, `personnel` (+ cv_url), `menu_items` (+ ingredients jsonb avec stockId), `orders` (+ order_type, tax_rate, stock_deducted), `order_items`, `reservations`, `suppliers` (+ invoice), `supplier_orders` (+ quantity/unit_price, expense_id, stock_id, received, stock_applied), `stocks` (+ initial_quantity, unit_cost), `stock_movements`, `expenses` (+ recurring_id), `recurring_expenses`, `incomes`, `special_events`, `deliveries`, `notifications`.

- Tout scoppé par `user_id = dataOwnerId`. Temps réel Socket.io par room = `dataOwnerId`.
- ⚠️ Modèle de confiance faible (le back fait confiance au `dbUserId` client) — OK en local, à durcir avant un déploiement public.
